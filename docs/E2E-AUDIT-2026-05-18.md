## Bugs fixed this round (all committed)

1. **user-service main class lived in the wrong package.** `com.vnshop.user_service` (with underscore) didn't match the entity package `com.vnshop.userservice`, so Spring Boot's default JPA scan missed every entity. Every request that touched the DB blew up with `UnknownEntityException: BuyerProfileJpaEntity`. The previous workaround (a `JpaConfig` with `@EntityScan` guarded by `@ConditionalOnBean(EntityManagerFactory.class)`) was a chicken-and-egg — the EMF needs entities found before it can be instantiated. **Fix:** moved the main class to the correct package, deleted the stub config. *(Commit b0fa68db.)*
2. **order-service was missing audit columns on `orders`, `sub_orders`, `order_items`.** `BaseJpaEntity` adds `created_at`/`updated_at`, but the V1 schema predates that refactor. Plain selects 500'd. **Fix:** V17 migration adds and backfills the columns. *(Commit 8b7bfb86.)*
3. **order-service outbox `payload` column is JSONB but Hibernate bound it as VARCHAR.** `columnDefinition = "jsonb"` only affects schema generation. **Fix:** added `@JdbcTypeCode(SqlTypes.JSON)`. *(Commit 8b7bfb86.)*
4. **order-service Kafka producer used StringSerializer for `OutboxEvent` records.** Every outbox publish failed with `ClassCastException`, so the `order_summary` projection table stayed empty even after a successful POST /orders. **Fix:** configured `JsonSerializer` in `application.yml`. *(Commit 8b7bfb86.)*
5. **Order numbers collided across restarts.** `Order.generateOrderNumber` uses an in-memory `AtomicInteger` that resets on every boot, so the first order created after a restart re-issued `VNS-yyyymmdd-00001` and the unique constraint tripped. **Fix:** millisecond-of-day prefix on the sequence. *(Commit 8b7bfb86.)*
6. **product-service Kafka producer same issue — ClassCastException on `ProductEvent`.** Every seller `POST /sellers/me/products` published nothing. **Fix:** `JsonSerializer` in product-service `application.yml`. *(Commit 7ef4cc98.)*
7. **Review + question DTOs required `buyerId` from the body and `text` for the review.** The standard FE shape sends `comment` and reads identity from the JWT, so every review/question 400'd. **Fix:** controllers now read buyer id from JWT, DTOs drop `buyerId`, `CreateReviewRequest.comment` accepts `text`/`body` as JSON aliases, `orderId` is optional. *(Commit 7ef4cc98.)*
8. **Gateway 401'd `/reviews` and `/recommendations` GETs for anonymous users.** The storefront fetches both on a product detail page before login. **Fix:** added `/reviews/**`, `/questions/**`, `/recommendations/**` to the public-GET allowlist alongside `/products`/`/search`. *(Commit 7ef4cc98.)*
9. **Gateway routing didn't forward `/auth/**` to user-service.** That path was permitted by SecurityConfig but never proxied. *(Already fixed in earlier session.)*
10. **`SubOrderJpaEntity.fromDomain` dropped the existing id on update.** Combined with `@OneToMany(orphanRemoval=true)` on the parent, every save of an order ran the orphan-removal path: existing sub-order rows were deleted and re-inserted with new BIGSERIAL ids. PUT /seller/orders/{id}/accept returned an id that became invalid as soon as the parent was saved, so the follow-up PUT .../ship 400'd with "subOrder not found". **Fix:** preserve `subOrder.id()` on `fromDomain`, so updates merge into the existing row. *(Commit f45fb4c0.)*

## Test-side robustness (2 fixes)

- **`GET /orders` is served by an asynchronous projection.** The order_summary table is populated by `OrderProjectionListener` consuming Kafka. After POST /orders returns 201, there's a 50–500 ms window before the projection catches up. The script now polls for up to 5s with a 250ms backoff before asserting; production callers should expect the same window. *(Commit 17892ae.)*
- **The projection exposes the order id as `orderId` (not `id`).** The script now accepts either field so it works against the projection's `OrderSummaryProjectionResponse` and the write-side `OrderResponse`. *(Commit 17892ae.)*

## Operational gotcha

- **Resilience4j circuit breakers latch.** When earlier 500s tripped the breaker, every subsequent request returned 503 (`Service temporarily unavailable`). After fixing root causes you must `docker compose restart api-gateway` before the next test run, or wait for the open→half-open transition (10s by default). The E2E run-time output makes this easy to spot.

## Pre-existing endpoints we did NOT exercise

The script focuses on the buyer/seller/admin happy-path "single day". The following exist but are out of scope here — they need their own scenario suites:

- VNPAY / MOMO payment intent creation + IPN callbacks (need a mock provider).
- Full saga compensation (cancel after partial fulfilment, return + refund).
- Admin dispute resolution beyond `GET /admin/disputes/open`.
- Coupon validate + apply (`/coupons/validate`).
- Cart guest mode + merge after login.
- Messaging WebSocket (`/ws/messaging`).
- Admin seller approval (`POST /admin/sellers/{id}/approve`).
- Notifications inbox.
- Shipping rate-quote (live endpoint flagged in pt2 handover).

A follow-up suite per scenario is the natural next step.
